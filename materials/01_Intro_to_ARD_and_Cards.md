# Chapter 1 — Introduction to ARD and `{cards}`

## 1.1 The Analysis Results Data (ARD) Concept

In traditional clinical trial programming, the output of an analysis is a formatted display—usually an RTF table, a PDF figure, or a text listing (TFL). While these documents are suitable for human review, they are highly problematic for automation, validation, and data reuse:

1. **Calculations are tied to presentation:** If you need to change a font, add a footnote, or modify a column layout, you often have to re-run the entire SAS or R program that performs the statistical calculations.
2. **Loss of numeric precision:** Numbers are rounded and formatted directly in the code (e.g., `Mean` is rounded to `74.4`). The raw, double-precision value (`74.380952...`) is lost unless the reviewer digs into the log or intermediate datasets.
3. **No machine-readable history:** There is no standard structure that tells a computer "this number represents the standard deviation of Age in the Safety Population for Xanomeline Low Dose."

**Analysis Results Data (ARD)** is a structured, tabular data format designed to store computed results and their metadata *before* formatting. In an ARD:
- Every calculated statistic (a mean, a count, a percentage, a p-value) lives on its own row.
- The raw, full-precision value is preserved alongside its display-ready formatted string.
- Full context (the variable, grouping variables, population, and statistical function) is explicitly documented in standard columns.

```
+-------------------------------------------------------------+
|                     ADaM Input Dataset                      |
+-------------------------------------------------------------+
                               |
                               | (Statistical Calculation)
                               v
+-------------------------------------------------------------+
|                Analysis Results Data (ARD)                  |
|  - One result value per row                                 |
|  - Full numeric precision + formatted text string           |
|  - Metadata (variable, grouping variables, statistics)      |
+-------------------------------------------------------------+
                               |
                               | (Presentation/Formatting)
                               v
+-----------------------+             +-----------------------+
|   gtsummary (HTML)    |             |    rtables (RTF/PDF)  |
+-----------------------+             +-----------------------+
```

## 1.2 The `{cards}` Package

The `{cards}` R package (maintained by Insights Engineering, a collaboration of Roche, GSK, Novartis, and others) is the foundational pharmaverse package for generating ARDs. 

Instead of writing custom, ad-hoc R code to calculate means or counts and then structuring them by hand, you use `{cards}` functions. These functions automatically output a data frame conforming to the CDISC-aligned ARD standard.

### Core Features of `{cards}`:
- **Consistent Output Structure:** Every `{cards}` function returns a data frame (or tibble) with the exact same columns.
- **Tidyverse-Friendly:** Integrates seamlessly with `{dplyr}` select helpers (`starts_with()`, `where(is.numeric)`, etc.).
- **Automatic Metadata Capture:** It captures the variable names, labels, and grouping factors from your input data automatically.
- **Format Agnostic:** The resulting ARD can be formatted into a table using `{gtsummary}`, `{rtables}`, or custom reporting tools.

## 1.3 The ARD Schema

Every ARD generated by `{cards}` has a standard set of columns. Understanding this schema is essential for working with ARD-driven pipelines.

| Column Name | Type | Description | Example |
|:---|:---|:---|:---|
| `group1` | `character` | Grouping variable name (if grouped) | `"TRT01A"` |
| `group1_level` | `any` | Grouping variable value/level | `"Placebo"` |
| `variable` | `character` | Variable being analyzed | `"AGE"` |
| `variable_grp` | `character` | Sub-group variable name (for categorical variables) | `"SEX"` |
| `variable_grp_level`| `any` | Sub-group level (for categorical variables) | `"M"` |
| `stat_name` | `character` | Short name of the statistic | `"mean"` |
| `stat_label` | `character` | Human-readable label of the statistic | `"Mean"` |
| `stat` | `list` (numeric/any)| Raw, unrounded numeric or object value | `75.2132...` |
| `stat_fmt` | `character` | Display-ready, formatted string | `"75.2"` |
| `context` | `character` | Originating function or context | `"ard_continuous"` |

*Note: The schema supports multiple grouping levels. If you group by treatment (`TRT01A`) and sex (`SEX`), `{cards}` will populate `group1 = "TRT01A"`, `group1_level = "Placebo"`, `group2 = "SEX"`, `group2_level = "M"`, and so on.*

## 1.4 Raw Value vs. Formatted Value

One of the most important concepts in ARD is the division between `stat` and `stat_fmt`:

- **`stat` (Raw Value):** This is stored as a list column in R, allowing it to hold any data type (numbers, matrices, or entire model objects). It preserves full numeric precision, which is critical for independent QC, auditing, and interactive exploration (e.g. plotting).
- **`stat_fmt` (Formatted Value):** This is a simple character column containing the rounded, formatted representation of the statistic (e.g. `"75.2"` or `"12.4%"`). This is what gets displayed in the final table.

By keeping both, ARD separates the **calculation step** (generating `stat`) from the **presentation step** (generating `stat_fmt`), resolving one of the biggest bottlenecks in clinical trial programming.

---

## Chapter 1 Summary

- **ARD (Analysis Results Data)** is a structured table storing statistical results and metadata.
- **`{cards}`** is the pharmaverse package used to generate ARDs from clinical data frames.
- **Raw values** (`stat`) preserve full numeric precision as list columns, while **formatted values** (`stat_fmt`) store character strings ready for display.
- **The ARD Schema** uses standard columns like `group1`, `variable`, `stat_name`, and `stat` to represent results in a machine-readable format.
